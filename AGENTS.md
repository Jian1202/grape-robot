# AGENTS.md

本文件是本仓库所有 Codex / AI Agent 的最高优先级项目内工作说明，作用范围为整个仓库。

> Codex：如果你刚进入这个项目，请先读完本文件，再按“必读顺序”读取项目事实和接口合同。不要因为某个函数名、ROS2 接口或硬件参数看起来合理，就假设它真实存在。

## 1. 给后续 Codex 的项目交接说明

这是一个基于 ROSLander 移动机械臂平台的葡萄视觉采摘项目，当前不是从零开始，也不是已经具备完整农业采摘闭环。

当前仓库中的 `track_and_grab.py` 已接入 YOLO 和 RGB-D 定位，并保留官方普通物体抓取流程；但它仍缺少或尚未可靠验证：

- 底盘进入机械臂有效工作距离；
- RGB 与深度空间对齐；
- 稳健的葡萄中心点与深度估计；
- 无目标时的主动搜索；
- 夹爪状态确认与异常归位；
- 面向悬挂葡萄的低速向下脱离；
- 抓取结果复检、超时、恢复和失败重试。

农业采摘旧案例只能提供流程思路，其中不少内容属于 ROS1、特定小车参数或未经当前实车验证的实现，禁止整体复制覆盖当前 ROS2 代码。

本项目使用 Codex 的原则是：

```text
高推理能力用于理解、设计、审查和纯算法实现
事实合同用于约束 ROS2 与硬件接口
测试和分级放行决定代码何时可以接触实车
```

任何时候都不能承诺“绝对不会出错”。必须确保未经证据和验证的代码无法直接获得机械臂、夹爪或底盘控制权限。

## 3. 语言和沟通要求

- 永远使用中文回复用户。
- 代码注释、项目文档和提交说明优先使用中文。
- 先报告结论和风险，再说明步骤。
- 不确定的内容必须明确写成“不知道”“未验证”或 `UNKNOWN`，不能用顺畅措辞掩盖缺口。

## 4. 事实状态

所有关键结论使用以下状态之一：

- `VERIFIED_REPO`：已在当前仓库源码或配置中找到；只证明仓库里存在，不证明实车可用。
- `DOCUMENTED_ROBOT`：项目文档记录为实车观察结果，但本次任务没有重新连接实车复核。
- `VERIFIED_ROBOT`：本次任务通过实机命令、接口输出或安全实验重新确认，并保存证据。
- `REFERENCE_ONLY`：来自厂商资料、官方示例或 ROS1 旧案例，只能参考。
- `INFERRED`：基于现有材料推断，尚未验证。
- `UNKNOWN`：没有可靠证据，禁止作为实现前提。

证据优先级不是简单地“文档写了就算完成”。涉及硬件动作时，优先使用当前实机接口输出、已安装接口定义和安全实验记录；仓库代码只能证明“代码试图这样调用”。

## 5. 禁止编造接口

不得编造或猜测：

- Python 包、import 路径、类、函数和方法；
- ROS2 node、topic、service、action、message 类型及字段；
- 坐标系名称、轴方向、单位和变换关系；
- 舵机 ID、pulse 范围、机械臂行程和夹爪开闭值；
- 相机内参、外参、RGB-D 对齐方式；
- IK 成功条件、返回字段和异常行为；
- 尚未实现的搜索、抓取、下拉、复检功能。

每个新增外部符号必须给出至少一个来源：

1. 当前仓库中的定义或真实调用位置；
2. 机器人已安装 ROS2 包中的源码或接口定义；
3. 本次保存的 `ros2 interface show`、`ros2 topic info -v`、`ros2 service type` 或 `ros2 node info` 输出；
4. 厂商原始源码，并明确标注是否为 ROS1 或不同版本。

找不到来源时必须停止相关实现，输出：

```text
UNKNOWN：缺少什么
为什么不能继续
需要在哪台设备执行什么只读命令
拿到结果后可以继续哪一步
```

## 6. 修改范围

- 任务开始时先执行只读校准：`git status -sb`、确认仓库根目录、阅读相关文件。
- 修改前列出准确文件白名单；不得顺手修改白名单外文件。
- 每次只解决一个可独立验证的问题，不进行无边界“整套重写”。
- `official_examples/` 是只读参考，除非用户明确要求更新镜像，否则禁止修改。
- `docs/source-materials/` 保存精选原始资料，禁止为了方便而改写原文件。
- 不覆盖组员未提交的改动，不使用破坏性 Git 命令清理工作区。
- 不把 ROS1 示例直接复制为当前 ROS2 实现。

## 7. 硬件安全边界

默认只允许离线分析和不驱动执行器的运行方式。

- `enable_arm`、`enable_grab`、`enable_detach`、`enable_base_align` 等动作开关默认必须为 `false`。
- 未经用户明确授权和现场人员确认，不运行会移动底盘、机械臂或夹爪的命令。
- 不直接向底层总线舵机话题发布未经验证的值。
- 不根据旧代码直接采用固定 pulse、位移、速度和坐标偏移。
- 所有硬件动作必须经过统一安全层，具备限幅、超时、失败退出、异常复位和停止能力。
- 视觉目标丢失、深度无效、RGB-D 未对齐、IK 无解、服务超时或状态反馈异常时，不得继续使用旧目标执行动作。
- 从检测模式进入实车动作前，必须按开发关卡逐级放行，不能跳级。

## 8. 代码设计原则

新增功能优先分为：

```text
纯算法层：目标筛选、中心点、深度过滤、状态转换，可离线测试
状态机层：流程、超时、失败和恢复，不直接发布硬件命令
ROS2 适配层：集中封装真实 topic/service/message
硬件安全层：限幅、互锁、急停、动作权限和反馈检查
配置层：保存已验证参数及单位，替代散落魔法值
```

只有 ROS2 适配层和硬件安全层可以直接接触控制接口。业务状态机不能在各处自行创建硬件 publisher/client。

## 9. 验证要求

根据改动风险至少完成以下检查：

- 文档：链接、路径、状态标签和事实一致性检查；
- Python：语法编译、导入检查和相关单元测试；
- ROS2：在正确环境中构建目标包并核对接口类型；
- 视觉算法：固定图片、深度样本或 rosbag 回放；
- 实车：检测-only、跟踪-only、夹爪单测、低速脱离、完整集成逐级进行。

无法在当前环境完成的验证必须明确列入“未验证项”，不能写成已经通过。

## 10. Git 与交付

- `main` 只保留已审查的稳定结果。
- Codex 默认使用 `codex/<任务名>` 分支。
- 每个提交和 PR 保持单一主题，不混入临时文件和无关修改。
- 提交前检查根 `README.md`、`docs/README.md` 和事实基线是否需要同步。
- 大型 rosbag、完整数据集、大量训练输出和不适合公开的厂商压缩包不提交普通 Git；使用私发、网盘、Release 或 Git LFS，并在仓库留下索引、版本和哈希。
- PR 必须填写 `.github/pull_request_template.md` 中的证据、风险、验证和回滚信息。
