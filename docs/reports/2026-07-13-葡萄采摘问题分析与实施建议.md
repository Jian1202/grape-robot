# 葡萄采摘问题分析与实施建议

日期：2026-07-13
范围：当前 ROS2 仓库、ROSLander 农业采摘案例、移动抓取案例、深度相机与机械臂课程资料

## 1. 结论

当前仓库并非完全缺少抓取能力，而是使用了不适合悬挂果实的官方基线。

现有 [track_and_grab.py](../../robot/grape_robot/code/track_and_grab.py) 主要继承普通空中物体抓取流程。农业采摘资料中的 `agricultural_picking.py` 已经包含底盘预对准、连续稳定检测、抓前修正、向下摘取、结果复检和失败重试，但这些农业专用逻辑尚未移植到当前 ROS2 实现。

因此，下一阶段不应只修改某个固定偏移，而应建立完整的葡萄采摘状态机。

## 2. 已确认的场景差异

通用抓取示例的任务是：

```text
发现普通物体 -> 夹取 -> 向上抬升 -> 搬运 -> 放置
```

悬挂葡萄的任务应是：

```text
寻找葡萄 -> 底盘进入机械臂工作距离 -> 视觉对准
-> 计算三维位置 -> 渐进接近 -> 夹紧
-> 向下脱离 -> 确认摘下 -> 搬运 -> 放置 -> 复位
```

当前代码在夹爪闭合后执行 `position[2] += 0.03`，属于向上抬升逻辑；农业采摘源码使用的是向下移动约 5cm。

## 3. 六个现象的判断

### 3.1 机械臂伸出不完全

这不一定是机械臂行程没有用完。可能原因包括：

- 小车停得太远或横向偏差过大；
- RGB 与深度坐标没有正确对齐；
- 手眼矩阵和夹爪 TCP 存在误差；
- 目标位置经过硬编码偏移后被算近；
- 目标位于逆运动学可达域边缘或无解；
- 舵机收到命令但因机械偏差、负载或摩擦没有到位。

农业采摘案例的处理顺序是先控制底盘左右平移和前后调距，将目标调整到约 0.30m，再让机械臂完成最后接近。当前项目应优先移植这一思路，而不是直接增加伸出量。

### 3.2 不知道葡萄藤高度

官方场地将果树横杆高度约束在 36cm，果实水平间距约 5-6cm、垂直间距约 5-7cm，因此原程序依赖固定场地，没有通用高度搜索。

当前项目需要先确定自己的使用条件：

- 固定教室场景：实测高度并配置安全扫描范围；
- 高度随机场景：增加机械臂俯仰扫描和底盘小角度搜索。

### 3.3 葡萄大小与中心点不准

当前 YOLO 检测框中心计算本身没有明显错误，但存在以下问题：

- `target_size` 计算后没有用于抓取决策；
- 多目标时固定选择最左侧目标；
- 使用固定的 `center_y - 40` 采样深度；
- RGB 像素直接用于深度图，空间对齐未完成验证；
- 10x10 深度区域使用均值，容易混入叶片或背景。

第一阶段建议使用 YOLO 框内部的稳健深度中位数、有效像素比例和多帧稳定性，不必立即引入完整 Open3D。

### 3.4 夹爪抓取后未归位

正常路径末尾存在夹爪复位命令，因此该现象还不能证明“代码没有复位”。需要排查：

- 动作线程是否中途异常；
- ID10 的 200、300、540、600 分别对应什么物理状态；
- 命令是否被其他节点覆盖；
- 舵机是否到达目标 pulse；
- 夹爪是否触发机械限位。

偏差手册建议 ID10 调试范围为 `[200,700]`，并指出超过 700 可能触碰机械限位。实车必须先建立 `open/close/hold` 三个经过验证的参数。

### 3.5 摄像头无目标时不会搜索

这是确定存在的逻辑缺失。当前代码无目标时只等待；农业采摘旧代码同样依赖固定场地和固定导航点，没有完整通用搜索。

需要自研：

```text
机械臂俯仰扫描 -> 左右扫描 -> 全部无目标
-> 底盘小角度旋转 -> 再扫描 -> 超时进入安全停止
```

### 3.6 应向下拉而不是向上抬

这是确定的策略问题。农业采摘源码在闭合夹爪后沿 Z 方向向下移动约 5cm，明确用于“向下拉，摘下小球”。

实车不能直接将 `+0.03` 改成 `-0.05` 后全自动运行。应从低速 2cm 开始，验证世界坐标方向、下方空间和 IK 可达性，再逐步扩大至 3-5cm。

## 4. 应从农业采摘源码移植的逻辑

收到私发的 `agripicking.zip` 后，重点阅读：

| 文件/函数 | 应整理的内容 | ROS2 移植目标 |
| --- | --- | --- |
| `config/config.yaml` | XYZ offset、导航点、目标距离 | 参数化配置，替代代码魔法值 |
| `mission_start.py` | 识别、导航、采摘的任务编排 | 系统级状态机 |
| `agricultural_picking.py::inter_cali_two` | 底盘横向和前后校准 | `BASE_ALIGN` 状态 |
| `agricultural_picking.py::judge_pick` | 目标筛选、深度稳定计数 | `LOCALIZE` 状态 |
| `agricultural_picking.py::pick` | 渐进修正、夹紧、向下摘取 | `APPROACH/GRASP/DETACH` 状态 |
| `agricultural_picking.py::depth_distance` | 目标附近深度计算 | 重写为稳健深度估计 |
| `agricultural_picking.py::place_functon_single` | 放置和下一次初始姿态 | `PLACE/RESET` 状态 |

## 5. 旧代码不能直接照搬的部分

- ROS1 接口与当前 ROS2 接口不兼容；
- RGB/深度只通过裁剪和比例缩放对应，不是完整空间标定；
- 深度区域仍采用均值；
- 部分框面积和类别拼写存在可疑实现；
- 视觉修正循环缺少明确超时；
- 舵机值和补偿值绑定特定小车；
- 没有通用无目标搜索；
- 压缩包包含大量与当前任务无关的第三方源码和模型文件。

应移植设计思路并重新实现，不应直接覆盖当前节点。

## 6. 推荐状态机

```text
IDLE
  -> SEARCH
  -> BASE_ALIGN
  -> ARM_TRACK
  -> LOCALIZE
  -> CHECK_IK
  -> PREGRASP
  -> VISUAL_REFINE
  -> CLOSE_GRIPPER
  -> PULL_DOWN
  -> VERIFY_DETACH
  -> PLACE
  -> RESET
```

失败统一进入：

```text
RECOVER -> 停止底盘 -> 张开夹爪 -> 回安全姿态 -> 重新搜索或人工确认
```

每个状态都必须有超时、失败原因和可观测日志。

## 7. 实施顺序

### 阶段 0：硬件与场地基线

1. 检查 1-5 号舵机中位姿态。
2. 检查夹爪中位张开是否约 2-3cm。
3. 实测 ID10 常用 pulse 对应状态。
4. 测量假葡萄高度、间距和小车初始距离。
5. 固定一个标准测试目标。

### 阶段 1：安全与日志

将一个 `enable_arm` 拆成：

```text
enable_tracking
enable_grab
enable_detach
enable_base_align
```

增加 IK 结果、目标三维坐标、舵机目标/反馈和状态转换日志。

### 阶段 2：视觉定位

- 去掉固定 `center_y - 40`；
- 使用框内中位数和离群值过滤；
- 验证 RGB 与深度对齐；
- 连续多帧稳定后才允许抓取；
- 目标选择加入完整度、深度有效性和可达性。

### 阶段 3：底盘预对准

先在单独安全开关下完成水平对准和前后调距，底盘稳定后才允许机械臂运动。

### 阶段 4：葡萄专用动作

实现预抓取、渐进接近、夹紧、低速向下脱离、成功验证、放置和复位。

### 阶段 5：主动搜索与完整任务

最后增加无目标扫描、导航点编排和多目标连续处理。
