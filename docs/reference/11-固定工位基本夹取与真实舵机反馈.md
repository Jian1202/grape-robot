## 3. 真实位置反馈合同

### 3.1 服务

```text
/ros_robot_controller/bus_servo/get_state
ros_robot_controller_msgs/srv/GetBusServoState
```

请求为`GetBusServoCmd[] cmd`。固定工位执行器对ID1–5、10分别设置：

```text
id=<舵机ID>
get_id=1
get_position=1
```

响应包含：

```text
bool success
BusServoState[] state
```

客户端必须同时验证：

1. `success=true`；
2. 返回状态数量等于请求数量；
3. 每项`present_id`只有一个值且等于请求ID；
4. 每项`position`只有一个值；
5. pulse有限且位于`[0, 1000]`；
6. 六个目标舵机均存在。

该服务调用只读取状态，不发送位置目标。

### 3.2 pulse到rad映射

当前控制器配置使用：

```text
radians_per_pulse = (240 / 360) * 2π / 1000
joint1–joint5 initial_pulse = 500
r_joint initial_pulse = 700
joint_rad = (initial_pulse - actual_pulse) * radians_per_pulse
```

舵机与关节映射：

| 舵机ID | 关节 |
| --- | --- |
| 1 | `joint1` |
| 2 | `joint2` |
| 3 | `joint3` |
| 4 | `joint4` |
| 5 | `joint5` |
| 10 | `r_joint` |

以上映射已写入
[basic_pick_plan.py](../../robot/grape_robot/code/basic_pick_plan.py)的纯函数并有单元测试。

### 3.3 为什么不能再用joint_states证明到位

厂商`ServoManager.get_position()`读取内部缓存；`set_position()`发送命令时会更新该缓存。
因此`/controller_manager/joint_states`可以反映控制器认为的目标，但不能回答舵机是否真的
到达目标。action成功和新到达的joint_states同时出现，仍不足以证明实物到位。

当前执行器保留joint_states用于：

- 检查ROS控制图是否存在且只有一个`controller_manager`发布者；
- inspect诊断显示；
- 与实际总线反馈进行人工对照。

启动前和每个action后的容差检查必须使用总线实际pulse转换后的rad。

## 5. Arm桌面工具互斥

Arm工具打开时会调用`enable_reception(False)`并直接控制舵机总线，此时禁止同时运行
ROS action执行器。

安全顺序：

1. 在Arm工具中点击“退出”，不要直接杀进程；
2. 等待至少5秒，让工具恢复`enable_reception(True)`；
3. 确认Arm工具已关闭；
4. 运行只读`inspect`；
5. 现场清空并确认后，才允许进入execute关卡。

Arm界面启动时滑块默认500，不代表真实位置。“角度回读”只读取pulse；回读行左侧按钮
可以把滑块同步到回读值。禁止误点“舵机回中”“手掰编程”或XYZ/pitch的IK `run`。

主数值框直接输入后必须按回车才会触发值变化；拖动主滑块也会立即发送。下方小滑块是
角度显示/联动区域，不能拿其显示值替代实际总线回读。

### 6.3 本项目如何参考预设

本项目只参考顺序：
