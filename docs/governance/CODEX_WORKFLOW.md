# 组员使用 Codex 的标准流程

本文件回答三个问题：组员如何让 Codex 快速理解项目、如何限制它不凭空补接口、如何用它完成深层复杂工作。

## 1. 从正确位置启动

必须在仓库根目录启动 Codex：

```powershell
cd E:\grape-robot
```

先确认当前目录能看到：

```text
AGENTS.md
README.md
docs/
robot/
tests/
```

标准约束文件是 `AGENTS.md`。不要只把某个 Python 文件单独交给 Codex，否则它可能看不到项目事实、接口状态和实车安全规则。

## 2. 第一句话怎样说

推荐组员先说：

```text
请先完整阅读根目录 AGENTS.md，并按其中必读顺序校准项目。
本轮先只读分析，不修改文件，不运行任何会驱动底盘、机械臂或夹爪的命令。
请先告诉我你确认的仓库根目录、当前分支、相关真实文件、接口来源和 UNKNOWN 项。
```

如果 Codex 没有先列证据就开始提出具体硬件函数、pulse 或 ROS2 接口，应立即让它停止并回到只读校准。

## 3. 一个复杂任务的六个阶段

### 阶段 A：只读调查

让 Codex 还原实际调用链，找定义、调用方、配置和失败路径，不允许编辑。

输出必须包括：

- 当前代码究竟做了什么；
- 外部接口来源；
- `VERIFIED_REPO / DOCUMENTED_ROBOT / REFERENCE_ONLY / UNKNOWN` 表；
- 哪些判断必须到机器人上验证；
- 风险和最小切入点。

### 阶段 B：接口取证

如果缺少实机接口，让 Codex只给出只读命令。组员在机器人上执行后，把完整输出保存到 `docs/evidence/`，再交给 Codex分析。

不要只发一张截断截图。优先保存文本、命令、日期、设备版本和执行环境。

### 阶段 C：设计评审

要求 Codex 在写代码前列出：

- 准备修改的准确文件；
- 新增类和函数签名；
- 谁调用它、它调用谁；
- 每个外部符号的来源；
- 状态机变化；
- 测试和回滚方案。

这一步适合使用高或 Ultra 推理档位，让模型深入理解整体系统，但仍不授权修改硬件接口。

### 阶段 D：小步实现

一次只实现一个独立能力，例如：

```text
目标选择
深度中位数过滤
连续帧稳定判断
服务调用超时
夹爪状态封装
搜索状态机中的一个状态
```

每轮给出文件白名单，禁止顺带重写整个节点。

### 阶段 E：独立审查

实现完成后，开启新的 Codex 任务或明确切换为审查角色，从零检查：

- 有没有无定义来源的符号；
- 有没有把 ROS1 当成 ROS2；
- 有没有单位、坐标轴或分辨率混淆；
- 有没有越界、空数组、NaN、旧目标复用；
- 有没有超时、停止、恢复和权限绕过；
- 默认安全开关是否仍为关闭。

同一个模型可以做实现和审查，但应使用独立上下文，避免它沿用自己先前的假设。

### 阶段 F：分级放行

严格按照[开发与实车放行关卡](DEVELOPMENT_GATES.md)从离线、回放、检测-only逐级进入实车。任何一级失败都退回修正，不能跳级。

## 4. 哪些内容可以让 Codex自主设计

在有测试的前提下，可以让 Codex 深入处理：

- 纯算法函数；
- 数据结构；
- 状态机组织；
- 日志、超时和错误处理；
- 单元测试、固定数据回放；
- 重复代码整理；
- 文档、接口表和实验记录格式。

## 5. 哪些内容必须由证据决定

Codex 不得自主决定：

- ROS2 topic、service、message 和字段；
- 外部包的函数签名；
- 坐标系方向和单位；
- 舵机 ID、pulse 和机械安全范围；
- 相机内外参和 RGB-D 对齐关系；
- 下拉方向、距离、速度和力；
- 急停、碰撞边界和安全姿态；
- 代码是否已经通过实车验证。

## 6. 推荐任务拆分

当前葡萄采摘工作建议拆成：

1. 实机接口快照与事实表；
2. 现有代码调用链和异常路径；
3. 目标选择与中心点；
4. 稳健深度与 RGB-D 对齐验证；
5. 状态机骨架；
6. ROS2 适配层；
7. 硬件安全网关；
8. 夹爪开闭与归位单测；
9. 底盘预对准；
10. 低速向下脱离；
11. 主动搜索；
12. 抓取复检与恢复；
13. 完整集成。

不要把以上内容合并成一个“请重写完整系统”的任务。

## 7. 组员与 Codex 的职责

| 角色 | 负责 | 不应代替谁 |
| --- | --- | --- |
| 资料整理组员 | 找原始文件、源码位置、版本和参数出处 | 不凭印象确认硬件事实 |
| ROS2 组员 | 在机器人上采集只读接口快照 | 不未经检查发布动作命令 |
| 视觉组员 | 准备固定测试集、深度样本和判定标准 | 不单凭单张成功图证明稳定性 |
| 机械臂组员 | 确认安全范围、方向、复位和现场保护 | 不让 Codex自行决定 pulse 与行程 |
| Codex | 阅读、对照、设计、实现、测试、审查和整理 | 不代替现场安全负责人放行实车 |
| 审查者 | 检查证据、风险、测试和回滚 | 不只看代码风格 |

## 8. 如何管理多轮 Codex 工作

- 一个任务对应一个明确目标和一个分支。
- 任务说明中写清“允许修改”和“禁止修改”。
- 让 Codex 把重要结论写回仓库，而不是只留在聊天记录。
- 新任务先读 `AGENTS.md` 和事实基线，不依赖上一次对话记忆。
- PR 中保留接口来源、验证结果和 `UNKNOWN` 项。
- 合并后删除过期分支，但不要删除实验原始证据。
- 事实改变时先更新事实基线和接口合同，再让后续 Codex继续开发。

## 9. 发现 Codex 开始猜测时怎么办

立即发送：

```text
停止实现。你刚才使用的外部函数、ROS2 接口或硬件参数没有给出定义来源。
请回到 AGENTS.md 的事实约束，只读搜索其真实定义。
如果找不到，请标记 UNKNOWN，给出需要执行的验证命令，不要继续补全代码。
```

不要通过反复让它“再想想”解决证据缺失；缺什么就去采集什么。

## 10. 一句话原则

```text
能力可以开到高或 Ultra，修改范围必须小，外部接口必须有源，硬件权限默认关闭。
```
