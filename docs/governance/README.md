# Codex 与项目治理入口

本目录用于规定“项目事实从哪里来、Codex 可以做什么、代码如何逐级进入实车”。它不是一般学习笔记，而是组员和 AI Agent 共同遵守的工作合同。

## 首次使用

组员第一次使用 Codex 前，按下面顺序阅读：

1. 仓库根目录 [AGENTS.md](../../AGENTS.md)；
2. [项目事实基线](PROJECT_FACT_BASE.md)；
3. [ROS2 接口合同](ROS2_INTERFACE_CONTRACT.md)；
4. [Codex 协作流程](CODEX_WORKFLOW.md)；
5. [开发与实车放行关卡](DEVELOPMENT_GATES.md)；
6. [提示词模板](PROMPT_TEMPLATES.md)。

启动 Codex 时应从仓库根目录进入。标准文件名是 `AGENTS.md`，不是 `AGENT.md`。Codex 会把根目录 `AGENTS.md` 作为项目级约束；如果从仓库外启动，或者只把单个文件发给模型，它可能看不到完整规则。

## 文件职责

| 文件 | 主要读者 | 解决的问题 | 何时更新 |
| --- | --- | --- | --- |
| `AGENTS.md` | Codex、所有开发者 | AI 的最高优先级边界、禁区和交付格式 | 协作规则或安全边界改变时 |
| `PROJECT_FACT_BASE.md` | 全组 | 哪些事实已经确认，哪些仍未知 | 实机验证、版本或硬件基线改变时 |
| `ROS2_INTERFACE_CONTRACT.md` | ROS2/机械臂开发者 | 当前依赖哪些接口、类型和证据 | 新增或修改任何 ROS2/外部接口时 |
| `CODEX_WORKFLOW.md` | 使用 Codex 的组员 | 如何拆任务、提问、审查和交接 | 工作流程优化时 |
| `DEVELOPMENT_GATES.md` | 现场负责人、代码审查者 | 什么条件下可以进入下一层实车测试 | 测试标准或安全流程改变时 |
| `PROMPT_TEMPLATES.md` | 所有组员 | 可复制的只读、设计、实现、审查提示词 | 发现新的常见误用时 |

## 维护原则

- `README` 负责导航，不重复堆积全部事实。
- 事实基线负责“现在知道什么”，不写希望和想象。
- 接口合同负责“代码怎样与外部系统通信”，不写未经确认的接口。
- 分析报告可以提出建议，但建议在验证前不能升级为 `VERIFIED`。
- 原始资料和 ROS1 示例是参考证据，不等于当前 ROS2 已实现。

## 最短工作闭环

```text
只读校准
-> 标注 VERIFIED / REFERENCE / UNKNOWN
-> 提出精确文件白名单和验收条件
-> 实现一个小改动
-> 离线测试或回放
-> 独立审查
-> 分级实车放行
-> 更新事实与接口合同
-> PR 合并
```
